Acciones de cocina y sus efectos en la BD
▶ Aceptar
UPDATE orders SET status = 'cooking'
+ INSERT order_status_history (changed_by='kitchen') → Realtime notifica al cliente 📡
✓ Listo
UPDATE orders SET status = 'ready'
+ INSERT order_status_history → Cliente ve "Tu pedido está listo" 🎉
📦 Archivar
UPDATE orders SET status = 'delivered'
+ UPDATE orders.completed_at = NOW()
Subscripción Realtime de la cocina
// cocina.html — escucha nuevos pedidos y cambios de estado
supabase.channel('orders-rt')
  .on('postgres_changes', { event: 'INSERT', table: 'orders' }, callback)
  .on('postgres_changes', { event: 'UPDATE', table: 'orders' }, callback)
  .subscribe()
Modo sin Supabase (Demo)
Si window.SUPABASE_CONFIG no está configurado, la cocina arranca en modo DEMO con 3 pedidos hardcodeados de ejemplo. No hay Realtime ni persistencia. Útil para mostrar la UI sin backend.

Modelo de seguridad de Supabase (por qué la anon key pública es segura)
✅ Anon Key — segura de ser pública
Diseñada para uso en el browser. Solo permite lo que las políticas RLS autorizan. Cualquiera puede ver el menú, crear pedidos, dejar calificaciones. Nada más. Equivalente a "clave de sólo-lectura parcial".
🚫 Service Role Key — NUNCA al frontend
Da acceso total sin restricciones a la base de datos, bypassea RLS. Solo en Supabase Dashboard. Nunca en el código, nunca en git.
